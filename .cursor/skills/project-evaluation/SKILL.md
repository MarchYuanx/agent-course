# 项目评估报告

对当前工作区项目进行多维度评估，输出优点、待改进点与打分，并将报告写入**固定目录**统一管理。

## 报告存放目录

- **路径**：项目根目录下的 `project-reports/` 文件夹。
- 若该目录不存在，先创建再写入报告。
- 单次报告文件名建议：`YYYY-MM-DD_project-evaluation.md`（按日期区分，便于管理）。

## 评估维度与打分

按下列维度评估并给出 1～10 分（可保留一位小数），再计算**综合分**（可对各维度取平均或加权）。

| 维度 | 说明 |
|------|------|
| 代码质量 | 正确性、边界与异常、可读性、命名与注释 |
| 结构与架构 | 目录与模块划分、职责清晰度、重复与耦合 |
| 文档与可读性 | README、注释、配置说明、上手成本 |
| 错误处理与健壮性 | 失败路径、错误信息、环境与配置校验 |
| 安全与敏感信息 | 注入风险、密钥与敏感数据、依赖安全 |
| 可维护性与扩展性 | 测试、依赖管理、配置外置、扩展点 |

## 报告结构模板

输出时按以下结构组织内容，并写入 `project-reports/YYYY-MM-DD_project-evaluation.md`：

```markdown
# 项目评估报告

**项目**：[项目根目录名或项目名]  
**评估日期**：YYYY-MM-DD

---

## 一、综合评分

| 维度 | 得分(1-10) | 简要说明 |
|------|------------|----------|
| 代码质量 | | |
| 结构与架构 | | |
| 文档与可读性 | | |
| 错误处理与健壮性 | | |
| 安全与敏感信息 | | |
| 可维护性与扩展性 | | |
| **综合分** | | |

---

## 二、优点

- 优点 1（可附具体位置或示例）
- 优点 2
- …

---

## 三、待改进点

- 改进点 1（尽量具体、可执行）
- 改进点 2
- …

---

## 四、总结与建议

（1～2 段简短总结，并给出 1～3 条优先建议。）
```

## 执行步骤

1. **浏览项目**：了解仓库结构、主要模块、依赖、README、关键代码与配置。
2. **按维度评估**：对照上述六个维度查看代码与文档，记录优缺点并打分。
3. **生成报告**：按模板填写综合评分、优点、待改进点、总结与建议。
4. **写入文件**：确保存在 `project-reports/`，将报告保存为 `project-reports/YYYY-MM-DD_project-evaluation.md`，并在对话中告知用户报告路径。

## 注意事项

- 评价基于当前工作区可见内容，不臆测未看到的仓库或环境。
- 待改进点尽量具体、可操作（如「在 xx 文件对 yy 增加校验」），避免空泛表述。
- 若项目根目录下已有 `project-reports` 的其它约定（如子目录、命名规则），可沿用并在报告中说明。
