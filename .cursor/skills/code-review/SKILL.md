# 代码审查

以资深工程师视角对当前 git 变更进行结构化审查，关注 SOLID、架构、可删除项与安全风险。默认仅输出审查结论，除非用户明确要求实现修改。

## 严重级别

| 级别 | 名称 | 说明 | 对应 |
|------|------|------|------|
| **P0** | 必须改 | 安全漏洞、数据丢失风险、正确性 bug | 🔴 合并前必须修复 |
| **P1** | 高 | 逻辑错误、明显 SOLID 违反、性能回退 | 🔴 合并前应修复 |
| **P2** | 中 | 代码坏味道、可维护性隐患、轻微 SOLID 违反 | 🟡 本 PR 或后续修复 |
| **P3** | 低 | 风格、命名、小建议 | 🟢 可选改进 |

## 工作流

### 1) 预检上下文

- 使用 `git status -sb`、`git diff --stat`、`git diff` 确定变更范围。
- 必要时用 `rg` 或 `grep` 查找相关模块、调用与契约。
- 识别入口、边界与关键路径（认证、支付、数据写入、网络）。

**边界情况**：
- **无变更**：若 `git diff` 为空，告知用户并询问是否审查已暂存变更或指定 commit 范围。
- **大 diff（>500 行）**：先按文件汇总，再按模块/功能分批审查。
- **混合关注点**：按逻辑功能分组，而非仅按文件顺序。

### 2) SOLID + 架构

- 加载 `references/solid-checklist.md` 获取具体提示。
- 关注：SRP（职责过载）、OCP（扩展需改核心）、LSP（子类破坏预期）、ISP（宽接口）、DIP（高层依赖具体实现）。
- 提出重构时说明**为何**能改善内聚/耦合，并给出最小、安全的拆分方案；若非平凡重构，给出增量计划而非大改。

### 3) 可删除项

- 加载 `references/removal-plan.md` 获取模板。
- 识别未使用、冗余或已关闭 feature flag 的代码。
- 区分：**可立即删除** vs **需计划后删除**，给出可执行的后续步骤与检查点。

### 4) 安全与可靠性

- 加载 `references/security-checklist.md` 获取覆盖项。
- 检查：XSS、注入（SQL/NoSQL/命令）、SSRF、路径穿越、密钥泄露、竞态、并发、未限流等。
- 同时说明**可利用性**与**影响**。

### 5) 代码质量

- 加载 `references/code-quality-checklist.md` 获取覆盖项。
- 检查：错误处理（吞异常、过宽 catch、缺失处理）、性能（N+1、热路径阻塞）、边界条件（null/空集合/数值边界）。
- 指出可能导致静默失败或生产事故的问题。

### 6) 输出格式

按以下结构组织审查结论：

```markdown
## 代码审查摘要

**审查文件**：X 个文件，Y 行变更
**总体结论**：[通过 / 需修改 / 建议]

---

## 发现项

### P0 - 必须改
（无或列表）

### P1 - 高
1. **[文件:行]** 简要标题
   - 问题描述
   - 建议修复

### P2 - 中
2. （跨节连续编号）
   - ...

### P3 - 低
...

---

## 可删除/迭代计划
（如有）

## 其他建议
（可选，非阻塞）
```

### 7) 下一步确认

给出结论后询问用户如何继续：

```markdown
---

## 下一步

共发现 X 项（P0: _, P1: _, P2: _, P3: _）。

**如何继续？**

1. **全部修复** - 实现所有建议
2. **仅修复 P0/P1** - 处理必须改与高优先级
3. **指定项修复** - 告诉我需修复哪些
4. **不修改** - 审查完成，无需实现

请选择或说明具体需求。
```

**重要**：在用户明确确认前，**不要**实施任何修改。本 skill 为「审查优先」工作流。
