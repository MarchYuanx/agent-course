# SOLID 坏味道提示

## SRP（单一职责）

- 文件承担无关职责（如 HTTP + DB + 领域规则混在一处）
- 大类/大模块低内聚，存在多种变更原因
- 函数编排大量无关步骤
- God 对象，对系统了解过多
- **自问**：「该模块因何单一原因而变更？」

## OCP（开闭原则）

- 新增行为需修改多处 switch/if
- 功能扩展需改核心逻辑而非扩展
- 无插件/策略/钩子等扩展点
- **自问**：「能否在不改现有代码的前提下增加新变体？」

## LSP（里氏替换）

- 子类对具体类型做检查，或对基类方法抛异常
- 重写方法弱化前置条件或强化后置条件
- 子类忽略或空实现父类行为
- **自问**：「能否用任意子类替换而不影响调用方？」

## ISP（接口隔离）

- 接口方法过多，多数实现者未使用
- 调用方依赖宽接口只为少量能力
- 接口方法空实现或 stub
- **自问**：「所有实现者是否都用到了所有方法？」

## DIP（依赖倒置）

- 高层逻辑依赖具体 IO、存储或网络实现
- 硬编码实现而非抽象或注入
- 导入链将业务逻辑与基础设施耦合
- **自问**：「能否在不改业务逻辑的前提下替换实现？」

---

## 常见代码坏味道（除 SOLID 外）

| 坏味道 | 表现 |
|--------|------|
| **过长方法** | 函数 > 30 行，多层嵌套 |
| **特性依恋** | 方法使用其他类的数据多于自身 |
| **数据泥团** | 同一组参数反复一起传递 |
| **基本类型偏执** | 用字符串/数字代替领域类型 |
| **霰弹式修改** | 一次变更需改多个文件 |
| **发散式变更** | 一个文件因多种无关原因而变更 |
| **死代码** | 不可达或从未调用的代码 |
| ** speculative generality** | 为假想未来而做的抽象 |
| **魔法数字/字符串** | 硬编码值无命名常量 |

## 重构启发

1. **按职责拆分，而非按大小** - 小文件也可能违反 SRP
2. **有第二次用例再抽象** - 避免过早抽象
3. **保持增量重构** - 先隔离行为再移动
4. **先保证行为不变** - 重构前加测试
5. **按意图命名** - 命名困难时，抽象可能有问题
6. **优先组合而非继承** - 继承带来紧耦合
7. **让非法状态不可表示** - 用类型约束不变量
